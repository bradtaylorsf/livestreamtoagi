## What Worked
- Two-stage actionable errors (missing pip package vs. missing Chromium binaries) wired into both `core/video/render_pipeline.py` and `scripts/render_simulation_video.py` with distinct exit codes (2/3) for orchestrator log clarity.
- CI `render-test` job caches both `~/.cache/uv` and `~/.cache/ms-playwright` keyed by playwright version, avoiding repeated browser downloads.
- Makefile targets pin `.venv/bin/python` and `.venv/bin/playwright` to bypass stale shims earlier on PATH — good defense against environment drift.

## What Failed
- One test-fix retry was needed before the suite went green (specific failure not captured in artifacts).

## Patterns
- Optional pip extra (`[render]`) + separate post-install step (`playwright install chromium`) documented in three places (pyproject, README, CLAUDE.md) keeps install paths discoverable.
- Preflight checks that return distinct exit codes per failure mode beat generic `exit(1)` for orchestrator-driven subprocesses.
- Probe multiple OS-specific cache dirs (macOS/Linux/Windows) plus `PLAYWRIGHT_BROWSERS_PATH` override to detect browser installation across environments.

## Anti-Patterns
- Worker (`core/video/worker.py`) launches subprocess with `stdout/stderr=DEVNULL`, so the carefully-crafted actionable errors are invisible unless a human runs the script directly — defeats the purpose of the clear messages.
- Leaving `playwright` as a commented note in `requirements.txt` risks drift; pyproject extras are the canonical source but the dual presence invites confusion.
