## What Worked
- End-to-end implementation landed in one pass: scenario YAML schema extension, MemorySeedConfig validation, MemorySeedApplier for all three modes (none/inherit/custom), CLI flags, and backend tests for each mode.
- All 2025 backend tests + 335 frontend + 211 website tests passed on first run with zero retries.
- Validation enforced at config-parse time (invalid agent_ids, missing inherit source, missing custom file) — failures are loud as required by the spec.

## What Failed
- Nothing blocking. Six INFO-level review findings (snapshot version mismatch noise, _apply_none over-writing for non-participating agents, double-wrap of inherited core memory, missing retention sweep for agent_energy_log, identical per-participant energy values, superset timeline payload) were left unfixed.

## Patterns
- Two-layer config (YAML schema + CLI override) with a single `MemorySeedConfig.from_dict` validator keeps mode-specific required-field rules in one place.
- Pre-existing snapshot infrastructure (MemorySnapshotImporter) was reused for both `inherit` and `custom` modes, so only a thin applier layer was new code.
- Seed-file fixtures (blank-slate.json, conflict-scenario.json) doubled as test inputs — same file format validated by `test_*_seed_file_loads_as_snapshot`.

## Anti-Patterns
- Shipping seed JSON with `version: 3` while `SNAPSHOT_VERSION = 1` causes a warning on every load — version constants should be bumped together or fixtures pinned to current version.
- Writing blank core_memory rows for *all* registered agents in `none` mode (instead of only `--agents`) pollutes the simulation namespace; scope writes to participating agents.
- Routing inherited core memory through the "new agent" branch wraps mature content inside `CORE_MEMORY_TEMPLATE`, double-wrapping it. Seeded restores should call `upsert_core_memory` directly.
- New append-only log tables (agent_energy_log) added without a corresponding retention policy will grow unbounded.
