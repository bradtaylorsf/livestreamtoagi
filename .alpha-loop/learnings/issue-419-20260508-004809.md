## What Worked
- Faction support implemented end-to-end in one pass: YAML schema parsing, scenario validation, orchestrator wiring, prompt injection via ContextAssembler, and adjacency boost in SpeakerSelector — all 1998 backend + 335 frontend + 211 website tests passed without retry.
- Reused existing alliance pair pattern (`set[frozenset[str]]`) for faction pairs in `speaker_selector.py`, keeping the adjacency-boost mechanism consistent and small (`_FACTION_ADJACENCY_BONUS = 0.15`).
- Reviewer caught and fixed a real gap: public `GET /api/simulations/{id}` was missing the new research fields that the acceptance criteria explicitly reference.

## What Failed
- Public GET endpoint in `core/public_routes.py` was initially overlooked — it builds its own response dict rather than reusing the admin Simulation model, so new fields didn't surface automatically.
- Two minor unfixed gaps remain: CLI `--hypothesis` is silently dropped when combined with `--sim-id` (existing-row branch skips `update_research_fields`), and no public POST simulations endpoint exists despite acceptance criteria phrasing.

## Patterns
- Pair-set pattern for relational scoring: when a feature defines groupings (alliances, factions), expand to all unordered pairs as `frozenset({a,b})` once at config load, then do O(1) membership checks in the hot path.
- Optional context section pattern: thread an optional `*_context: str | None` param through `ContextAssembler.assemble()`, track it in `_track()`, and append to `system_sections` only when truthy. Keeps prompt assembly composable and prevents empty headers.
- Duck-typed config objects: `getattr(f, "members", None) or f.get("members", [])` lets orchestrator accept both dataclass and dict faction representations without tight coupling to YAML parser internals.

## Anti-Patterns
- Hand-rolled response dicts in public routes drift from admin models. When adding fields to a Pydantic model used by admin routes, grep for parallel public endpoints that construct dicts manually — they won't auto-update.
- Two code paths for simulation creation (pre-created row via `--sim-id` vs. fresh create) with research fields only handled in one path. Either branch should call the same persistence helper or hypothesis should be passed through both.
