## What Worked
- Implementation delivered the `/simulations/live` Wall of Simulations page with grid rendering, filter tabs (All/Running/Recent/Featured), and 5s polling fallback — all acceptance criteria met
- Backend additions (PATCH /simulations/{id}, public auth magic-link flow, submission caps) landed with comprehensive test coverage; full suite passed first try (2110 backend, 335 frontend, 280 website)
- SimulationWall component was tested in isolation (20 tests) covering tab filtering, polling, and fallback rendering

## What Failed
- Nothing blocking — all tests passed on first run with zero retry cycles

## Patterns
- Adding `completed_within_hours` and `is_featured` query params to the existing list endpoint kept the Wall page on a single REST call rather than introducing a new endpoint
- Joining `users` table with `split_part(email, '@', 1)` for `submitter_display_name` gave anonymous-by-default behavior with opt-in display via auth, no schema migration for display names
- Magic-link verify checks `AUTH_JWT_SECRET` config *before* consuming the token, so misconfigured deploys don't burn user tokens

## Anti-Patterns
- SimulationWall reused a single `cancelledRef` across tab-switch effects, creating a race where stale fetch results can populate the new tab briefly — should use AbortController or per-fetch token instead
- Built the share-as-challenge endpoint + client + tests but left no UI surface to invoke it; backend-first work without a tracked UI follow-up risks the feature being technically complete but user-inaccessible
- New `/simulations/live` page not linked from main Navigation — discoverability gap not caught because acceptance criteria didn't explicitly require nav integration
