## What Worked
- Implemented full public submission endpoint with all four cap types (per-submission, lifetime, concurrent, daily) plus happy-path and per-cap tests (38/38 backend tests pass).
- Bundled dependency #422 (magic-link auth) into the same run, including UserRepo/MagicLinkTokenRepo, JWT cookie session, and email rate limiting.
- Reviewer caught and fixed a real bug: magic-link token was being consumed before AUTH_JWT_SECRET check, burning the link on misconfigured deploys.

## What Failed
- TOCTOU race left unfixed between `count_active_for_user()` and the INSERT — concurrent submissions could both pass the concurrent=1 cap.
- `tests/backend/test_migrations.py` ALL_TABLES and DROP list not updated for new `users` / `magic_link_tokens` tables from migration 041.
- `secure=True` hardcoded on user_session cookie blocks magic-link flow on plain http://localhost dev (matches admin pattern, but undocumented friction).

## Patterns
- Token-consumption ordering: validate all required config (signing secrets, etc.) *before* consuming single-use tokens, so misconfig fails closed without destroying user state.
- Bundling tightly-coupled issues (auth + first authenticated endpoint) into one PR keeps the dependency chain testable end-to-end.

## Anti-Patterns
- Adding new tables in a migration without updating the integration-test schema fixtures (ALL_TABLES, DROP guards) — silent drift that won't fail until much later.
- Relying on application-level count-then-insert for uniqueness caps; a partial unique index is the only race-free enforcement.
