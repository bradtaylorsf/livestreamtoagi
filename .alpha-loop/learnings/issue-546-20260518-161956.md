## What Worked
- Trace ID added as a purely additive, optional field on both request/response envelopes with a minor protocol bump (1.0→1.1) and major-only version gating, so 1.0 peers stay wire-compatible — no contract break.

## What Failed
- Nothing functional. Reviewer caught one real defect: caller-controlled string fields (trace_id, request_id, agent_id, service/method) were written verbatim into the single-line `key=value` log record on both Python and Node sides — a log-injection / line-corruption vector that directly undermines the "one trace ID greps cleanly" property. Fixed with a shared `_safe_str`/`_safeStr` sanitizer at the single formatting chokepoint plus a regression test.

## Patterns
- Sanitize caller-controlled strings at the single log-formatting chokepoint (collapse whitespace/control chars, cap length) whenever building delimited single-line log records — the grep-correlation guarantee is itself a security property.

## Anti-Patterns
- Rendering authenticated-but-untrusted protocol fields into structured logs without sanitization — newline/space injection can forge or split log records and break cross-system correlation.

## Suggested Skill Updates
- `security-analysis`: add a log-injection check for caller/peer-controlled fields rendered into single-line delimited log records (newline/space splitting, record forgery, oversized-field flooding), and treat correlation-ID integrity as a security property, not just an observability nicety.
