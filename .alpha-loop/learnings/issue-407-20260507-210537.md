## What Worked
- Backend plumbing of optional `simulation_id` query param across `/api/agents/*`, `/api/conversations`, and `/api/evals/runs`/`latest` endpoints, with unscoped requests aggregating across all sims as specified.
- Client-side Zustand store with `sessionStorage` persistence (`simulation-store.ts`) plus `useCurrentSimulationId` hook drove consistent reads/writes across pages; tests passed on first run with zero retries.
- Reviewer caught and fixed a regression in `AgentDetailClient` where initial tab loads ignored the hydrated `simulation_id`, restoring the verification flow in the AC.

## What Failed
- Partial AC coverage: `/evals/summary`, `/evals/history`, and `/evals/categories` were not updated to accept `simulation_id`, leaving the "consistently" clause of the AC literally unmet despite default behavior matching intent.
- Hydration mismatch risk in `useCurrentSimulationId` — server renders `null` while client first render has the persisted UUID; works via React reconciliation but emits a warning.

## Patterns
- Pair a small Zustand store + `sessionStorage` for cross-page selection state, exposing a hook for components and a plain getter for non-React call sites (initial fetches, query builders).
- When adding an optional filter param, default to "unscoped = aggregate across all" rather than silently falling back to a magic ID like `LIVE_SIMULATION_ID`; makes the param's absence semantically meaningful.
- Extract a private `_build_agent_profile` helper when an endpoint and its detail variant share assembly logic but differ only in scoping inputs.

## Anti-Patterns
- Hydrating filter UI state from `sessionStorage` but not threading that same value into the initial data fetch — UI and data diverge on first render until the user re-triggers.
- Sweeping AC checklist items as "done" when only a subset of an endpoint family is updated; "consistently" in an AC means every endpoint in the listed glob, not just the ones the new UI happens to call.
- Path-traversal validation for user-supplied filenames inline in the route handler is fine here, but tends to drift; future similar features should centralize the `resolve().relative_to(allowed_dir)` check in a helper.

## Suggested Skill Updates
- `code-review`: add a check for "filter state hydrated from storage but not passed to initial fetch" — common Next.js/React pitfall worth catching automatically.
- `implementation-planning`: when ACs say "all X endpoints", require the plan to enumerate the endpoint family explicitly so partial coverage is caught at planning time, not review time.
