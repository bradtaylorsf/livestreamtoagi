## What Worked
- Tests passed on first run (0 retries) across pytest (2089), frontend vitest (335), and website vitest (211).
- Idempotent video render claim via DB state machine (unrendered → rendering → done/failed/skipped) prevents duplicate work.
- Graceful failure modes implemented per acceptance criteria: no transcripts marks `skipped`, render errors mark `failed` rather than crashing.

## What Failed
- Cross-issue integration gap: completion email sends before render finishes, so users never receive the video link (spec-allowed but degrades UX).
- `/simulations/{id}/replay` frontend page referenced by the Playwright render script does not exist in this PR (deferred to E2-14).
- Playwright is imported but not declared in `pyproject.toml`, so production renders fail with `RenderError('playwright is not installed')` until deps are added.

## Patterns
- DB-backed claim pattern (`UPDATE ... WHERE video_render_status IS NULL RETURNING`) for idempotent background-job dispatch.
- Env-driven render config with sensible defaults (`MAX_VIDEO_RENDER_MINUTES`, `VIDEO_STORAGE`) keeps the worker testable without infra.
- Mock-the-render testing strategy: assert queue/state transitions, not ffmpeg/Playwright output, keeps unit tests fast and deterministic.

## Anti-Patterns
- Detaching a subprocess and immediately notifying users of "completion" — the user-visible artifact (video URL) isn't ready yet.
- Importing optional heavy deps (Playwright) lazily without declaring them; tests pass but production silently fails.
- Building a renderer that depends on a frontend route owned by another ticket without gating or stub — creates a runtime cliff.
