# Implementer Agent

You implement GitHub issues by writing code, tests, and committing changes.

## Pre-Implementation Checklist

Before writing any code:

1. **Read the issue thoroughly** — understand acceptance criteria before starting
2. **Check existing code** — read related files to understand conventions and patterns
3. **Plan before coding** — identify which files to create/modify

## Environment Rules

### CRITICAL: Docker Service Health Gate

Before running ANY integration test or database-dependent test, you MUST verify Docker services are healthy:

```bash
docker compose up -d && bash scripts/check-services.sh
```

All checks must pass before proceeding. If services are unhealthy:
- Check for port conflicts: `lsof -i :6381` (Redis), `lsof -i :5434` (PostgreSQL), `lsof -i :3100` (Langfuse)
- Check Docker logs: `docker compose logs <service>`
- Fix the root cause before retrying tests

**Do NOT retry tests if the first failure looks like a connection/environment error. Fix the environment first.**

### Environment Variables

- Secrets used in Docker services must be cryptographically strong (32+ characters) — never use placeholder values like `dev-salt-change-me`
- Use `${VAR:-default}` patterns in compose files for developer-overridable ports
- Default ports for this project: Redis=6381, PostgreSQL=5434, Langfuse=3100

## Python Code Patterns

### FastAPI

- Place the FastAPI app in `core/main.py`, not root `main.py`
- Use `asynccontextmanager` lifespan pattern for startup/shutdown, not deprecated `on_event`
- Health endpoints should report per-dependency status and return "degraded" on partial failure

### Database

- Use parameterized queries — never use f-strings or string interpolation in SQL
- Use `CREATE TABLE IF NOT EXISTS` and `ON CONFLICT DO NOTHING` for idempotent migrations
- Register pgvector codec gracefully (catch `ValueError` for missing extension)

### Async Generators

- **Always place cleanup logic (cost logging, tracing, resource release) inside `finally` blocks**, not after them
- Async generator consumers may terminate early via `GeneratorExit` — code after `try/finally` will be skipped silently

### Dependencies

- Use `pyproject.toml` as the single source of truth for dependencies
- Use `[project.optional-dependencies]` for dev deps
- Do NOT maintain a separate `requirements.txt` — generate it from pyproject.toml if needed
- Track `uv.lock` in git for reproducible installs

## Testing Rules

### Unit Tests

- **NEVER instantiate `TestClient(app)` at module level** — this triggers FastAPI lifespan hooks (DB/Redis connections) at import time, causing failures without Docker
- Always create `TestClient` inside test functions or fixtures
- Mock FastAPI lifespan dependencies (db, redis) in unit tests to avoid requiring Docker

```python
# ❌ BAD: Module-level instantiation triggers real connections
from core.main import app
client = TestClient(app)  # This runs lifespan at import!

def test_health():
    response = client.get("/health")
    assert response.status_code == 200

# ✅ GOOD: Create client in fixture with mocked dependencies
import pytest
from unittest.mock import AsyncMock, patch

@pytest.fixture
def client():
    with patch("core.main.get_db", return_value=AsyncMock()):
        with patch("core.main.get_redis", return_value=AsyncMock()):
            with TestClient(app) as c:
                yield c

def test_health(client):
    response = client.get("/health")
    assert response.status_code == 200
```

### Integration Tests

- Always run `docker compose up -d && bash scripts/check-services.sh` before integration tests
- Use `pytest.mark.skipif` when required env vars are missing, rather than failing or mocking
- Use `pytest.mark.integration` to separate integration tests from unit tests

### Test Data

- Use factory functions for test data, not hard-coded values
- Use unique identifiers (timestamps, UUIDs) to avoid test conflicts
- Clean up test data in teardown

## Code Quality

- **No dead code** — only create Pydantic models, functions, and classes that are used by existing code paths
- **No placeholder strings** — don't add template placeholders like `{injected_from_sentinel}` without the injection system
- **Use specific types** — prefer `list[str]` over `Any` for typed JSONB columns
- **Name constants accurately** — `MAX_RETRIES=3` means 3 retries (4 total attempts), `MAX_ATTEMPTS=3` means 3 total attempts

## Commit Rules

- One logical change per commit
- Commit message explains WHY, diff shows WHAT
- Use conventional commits: `feat:`, `fix:`, `test:`, `refactor:`, `docs:`, `chore:`
- Reference issue number: `feat: add health endpoint (closes #1)`
- **After retry loops**: each retry commit must describe what specifically changed, not generic "fix: resolve verification failures"

## Retry Discipline

When a test or verification step fails:

1. **Read the error output carefully** — diagnose the root cause
2. **Distinguish environment failures from code failures** — if it's a connection error, fix Docker/services first
3. **Log what failed and why** before making changes
4. **If the same error repeats**: check environment health, don't just retry the same code
5. **Never consume a retry on an environment issue** — fix the environment, then retry

# Testing Patterns Skill

## Trigger
When writing tests or implementing features that need tests.

## Python (pytest) Patterns

### Test Structure

```python
import pytest
from unittest.mock import AsyncMock, patch

class TestModuleName:
    """Tests for module_name."""

    def test_expected_behavior_when_given_input(self):
        # Arrange
        input_data = create_test_input()

        # Act
        result = function_name(input_data)

        # Assert
        assert result == expected_output

    def test_raises_when_given_invalid_input(self):
        with pytest.raises(ValueError):
            function_name(None)
```

### FastAPI Test Rules

**CRITICAL: Never instantiate TestClient at module level.**

```python
# ❌ BAD: Triggers lifespan (DB/Redis connections) at import time
from core.main import app
from fastapi.testclient import TestClient
client = TestClient(app)  # Runs at import — fails without Docker!

# ✅ GOOD: Create in fixture with mocked dependencies
@pytest.fixture
def client():
    with patch("core.main.get_db", return_value=AsyncMock()):
        with patch("core.main.get_redis", return_value=AsyncMock()):
            with TestClient(app) as c:
                yield c

def test_health(client):
    response = client.get("/health")
    assert response.status_code == 200
```

### Integration Tests

```python
import pytest
import os

# Skip if Docker services not available
pytestmark = pytest.mark.skipif(
    not os.getenv("DATABASE_URL"),
    reason="DATABASE_URL not set — Docker services required"
)

class TestDatabaseIntegration:
    async def test_connection(self, db_pool):
        async with db_pool.acquire() as conn:
            result = await conn.fetchval("SELECT 1")
            assert result == 1
```

### Pre-Test Environment Check

Before running integration tests, always verify services:
```bash

```

All 5 checks must pass (Redis, PostgreSQL, pgvector, pg_trgm, Langfuse) before proceeding.

### Async Generator Testing

```python
# ✅ Test that cleanup runs on early termination
async def test_streaming_cleanup_on_early_exit():
    cleanup_ran = False

    async def stream():
        nonlocal cleanup_ran
        try:
            yield "chunk1"
            yield "chunk2"
        finally:
            cleanup_ran = True  # Must be in finally, not after try/finally

    gen = stream()
    await gen.__anext__()  # Get first chunk
    await gen.aclose()      # Simulate early termination
    assert cleanup_ran
```

### Mocking

- Mock external dependencies (DB, Redis, HTTP clients) in unit tests
- Use `unittest.mock.AsyncMock` for async dependencies
- Use `pytest.fixture` for test setup, not module-level code
- Reset mocks between tests with `autouse` fixtures if needed

## TypeScript (Jest) Patterns

### Test Structure

```typescript
describe('ModuleName', () => {
  describe('functionName', () => {
    it('should do expected behavior when given input', () => {
      // Arrange
      const input = createTestInput();

      // Act
      const result = functionName(input);

      // Assert
      expect(result).toEqual(expectedOutput);
    });

    it('should throw when given invalid input', () => {
      expect(() => functionName(null)).toThrow();
    });
  });
});
```

### Rules (Both Languages)

#### Naming
- Describe behavior, not implementation
- Good: `test_returns_404_when_user_not_found` / `should return 404 when user not found`
- Bad: `test_get_user_by_id` / `test getUserById`

#### Isolation
- Each test is independent (no shared mutable state)
- Use fixtures/`beforeEach` for setup
- Clean up after tests (close connections, clear timers)

#### What to Test
- Happy path (expected input -> expected output)
- Error cases (invalid input, missing data, network failures)
- Edge cases (empty collections, null values, boundary conditions)
- Integration points (API endpoints)

#### What NOT to Test
- Third-party library internals
- Type system checks (the compiler does this)
- Simple getters/setters with no logic

#### Anti-Patterns to Avoid
- No hardcoded test IDs that depend on database state
- No tests that depend on execution order
- No module-level side effects (TestClient, DB connections)
- No sleeping/waiting for arbitrary durations in tests

# Git Workflow Skill

## Trigger
When creating branches, writing commits, or creating PRs.

## Branch Naming

```
agent/issue-{number}     # automated loop branches
feat/{description}       # feature branches
fix/{description}        # bug fix branches
```

## Commit Conventions

Use conventional commits:

```
feat: add health check endpoint (closes #1)
fix: resolve test failures for #3
test: add unit tests for runner module
refactor: simplify prompt building
docs: update CLAUDE.md with new skills
chore: update dependencies
```

Rules:
- Lowercase, no period at end
- Reference issue number when applicable
- One logical change per commit
- Commit message explains WHY, diff shows WHAT

## Retry Commit Hygiene

When a task requires multiple retries:

1. **Each retry commit must have a distinct, specific message** describing what changed:
   - Good: `fix: mock lifespan dependencies in unit tests to avoid Docker requirement`
   - Good: `fix: move cost logging into finally block for early generator termination`
   - Bad: `fix: resolve verification failures` (repeated 3 times)
   - Bad: `fix: address review feedback` (what feedback?)

2. **Before opening the PR**, review commit history. If retry commits are vague or redundant, consider squashing them into a single descriptive commit.

3. **Annotate retry causes**: When a retry is triggered, note in the commit message what specifically failed in the previous attempt.

## PR Workflow

1. Branch from the target branch (master or session branch)
2. Implement + test + commit
3. Push branch
4. Create PR with: summary, test results, review report
5. Link PR to issue with `closes #N`

## Never Do

- Force push to master/main
- Commit directly to master/main
- Commit .env files or secrets
- Create merge commits (use squash merge)
- Include unrelated file changes in feature branches (e.g., `frontend/package-lock.json` in a backend PR)
