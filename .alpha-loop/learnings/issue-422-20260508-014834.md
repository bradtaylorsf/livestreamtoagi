## What Worked
- Magic-link auth implemented end-to-end (request, verify, logout, /me) with 38 passing backend tests covering email validation, JWT lifecycle, rate limiting, and cookie flow.
- Reviewer caught and fixed a real footgun: AUTH_JWT_SECRET check moved before token consumption so a misconfigured deploy doesn't burn the user's single-use link.
- Token storage uses SHA-256 hashing (raw token never persisted), and rate limiting (5/hr per email) degrades gracefully when Redis is unavailable.

## What Failed
- Nothing blocking. Three known unfixed issues documented: TOCTOU race on per-user concurrent/lifetime caps, hardcoded `secure=True` cookie breaks http://localhost dev, and `test_migrations.py` ALL_TABLES list not updated for new `users`/`magic_link_tokens` tables.

## Patterns
- Check configuration prerequisites (secrets, env vars) BEFORE consuming single-use resources (tokens, nonces) — fail closed without destroying state.
- Always-200 response on magic-link request to avoid leaking account existence; surface delivery failures only via 503 when send actually errors.
- Hash single-use tokens at rest (store hash, compare hash) so a DB leak doesn't yield usable links.

## Anti-Patterns
- Updating migrations without updating the migration test's ALL_TABLES / DROP list — integration tests pass but rollback coverage silently regresses.
- Copying `secure=True` cookie config from admin auth into a public-facing flow without considering the dev-server (HTTP) UX impact.
- Doing existence/cap checks in app code without a DB-level partial unique index when concurrent submissions from the same actor are possible.

## Suggested Skill Updates
- `security-analysis`: add a check item for "single-use token consumed before all preconditions verified" (auth secrets, signing keys, downstream service health).
- `testing-patterns`: when adding new tables in a migration, update the migration test's table list in the same PR — add this to the test-writing checklist.
